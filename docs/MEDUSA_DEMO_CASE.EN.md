# Medusa Demo Case Study

This case study is based on the `medusa-demo` `.claude/skills/skills-seed-skills` directory generated with the default `sync` flow. The point is not that Skills Seed can write a project summary. The point is that it can turn business constraints, module boundaries, entry methods, and validation commands scattered across a large TypeScript monorepo into a project-level Skill that an Agent can load by task.

Example output: [medusa-demo/.claude/skills/skills-seed-skills](https://github.com/silaswei-io/medusa-demo/tree/develop/.claude/skills/skills-seed-skills)

## Generated Output

After the default sync, the entry file is `SKILL.md`, and deeper content is split under `references/`:

```text
.claude/skills/skills-seed-skills/
├── SKILL.md
└── references/
    ├── project-overview.md
    ├── project-spec.md
    ├── business-methods.md
    ├── common-utils.md
    ├── modules.md
    └── patterns/
        ├── api.md
        ├── business.md
        ├── concurrency.md
        ├── config.md
        ├── database.md
        ├── error.md
        ├── middleware.md
        ├── naming.md
        ├── structure.md
        └── utils.md
```

The generated entry Skill learned 92 patterns across 10 categories, with average confidence around 90%. The project profile identifies Medusa's monorepo architecture, workflow orchestration, module services, API routes, configuration, events, remote query behavior, and validation commands, then routes detailed information into the relevant reference files.

## Project-Level Value

Task Skills in medusa-demo, such as `writing-docs`, `reviewing-prs`, `triaging-issues`, and `writing-releases`, describe fixed workflows: how to write MDX documentation, review a PR, or emit a review decision. They focus on "what process should be followed for this task type."

The generated `skills-seed-skills` works at a different layer: the current codebase's project semantics. It tells the Agent:

- Which references to read before changing APIs, business flows, configuration, or external dependencies.
- Which rules are backed by real code evidence, and which items are only navigation observations.
- Where Medusa conventions such as workflows, step compensation, module services, route loading, DTO naming, and error handling appear in files and patterns.
- Which validation command fits a change scope, such as `yarn validate:http-types` for API / contract changes or `yarn test` for persistence and query changes.

In short, task Skills provide process playbooks, while the Skill generated by Skills Seed provides a project map and constraint index extracted from code. Use `writing-docs` for documentation work and `skills-seed-skills` when a change touches Medusa's real module boundaries, business state machines, generated artifacts, or validation scope.

## How An Agent Uses It

For a task like "add an API field", the entry `SKILL.md` routes the Agent to `project-spec.md`, API patterns, and structure patterns instead of forcing a full repository read. The Agent can see how interface contracts, generated artifacts, transformation layers, and validation commands relate, which reduces the chance of changing implementation only while missing types or generation checks.

For order, payment, inventory, fulfillment, or similar business flows, the entry routes the Agent to the business pattern map first, then into the domain-specific reference. That gives the Agent source-backed context for state transitions, locks, compensation logic, and event ordering before it edits code.

For utilities, configuration, error handling, or middleware, the Agent only needs the matching category reference. The entry document intentionally stays small so the 90+ learned rules are not loaded all at once.

## Core Takeaways

| Value | medusa-demo example |
|---|---|
| Learns from real code | Patterns come from the current TypeScript monorepo's modules, workflows, routes, config, and test commands |
| Loads context on demand | `SKILL.md` routes tasks; details stay in `references/` |
| Keeps evidence and boundaries | Rules distinguish hard constraints from reference observations and tell Agents to trust current code |
| Fits complex repositories | Handles multi-module monorepos, cross-module orchestration, generated artifacts, and validation commands |
| Works with task Skills | Task Skills handle process; Skills Seed supplies project semantics and code constraints |

## Why This Shows Skills Seed's Value

The hardest project knowledge to pass to an Agent is usually not what is already written in a README. It is the implicit convention distributed across code, directories, names, error branches, validation commands, and historical changes. The medusa-demo output shows that a default Skills Seed sync can turn those conventions into a local Skill that is routeable, refreshable, and shareable.

This lets an Agent get project-level judgment before editing: which module owns the task, which references to read, which rules have evidence, and which commands validate the change. For a team, that is more stable than re-explaining context every session and easier to keep current than a long hand-written project guide.
